## Welcome to Dungeon 1

You just walked into the very first dungeon of DSA Quest. Before we fight anything: Arrays and Hashing is the foundation for everything else. Almost every problem you'll ever solve starts here.

Today's boss: **The Biryani Token**. Your weapon to unlock: **the hash set**.

---

## The story

Tonight is the biggest wedding feast of the year. We are talking 5,000 guests and the smell of fresh biryani everywhere. You just got hired to manage the food tokens at the gate.

You show up early and the manager hands you a stack of 5,000 numbered tokens. He looks worried.

"One of our printers had a glitch last week. Some of these tokens might be duplicates. If two guests show up with the same number, we'll run out of food and there will be a riot. Before doors open in ten minutes, tell me — is there a duplicate anywhere in this stack?"

Ten minutes. Five thousand tokens. You need a plan.

---

## The naive way (please don't)

Your first instinct is to compare every token with every other token. You pick up token #1 and check it against tokens #2, #3, #4... all the way to #5000. Then you pick up token #2 and check it against #3, #4... and so on.

```python
def has_duplicate_slow(tokens):
    for i in range(len(tokens)):
        for j in range(i + 1, len(tokens)):
            if tokens[i] == tokens[j]:
                return True
    return False
```

Let's count the damage.

With 5,000 tokens, that's roughly `5000 × 5000 / 2 = 12.5 million` comparisons. If each comparison takes one microsecond, you're done in 12 seconds. Okay, survivable.

But what if the party is a mega-event next month with 100,000 guests? Now it's `100,000 × 100,000 / 2 = 5 billion` comparisons. That's not ten minutes. That's an hour and a half.

This is O(n²). It works. But it scales like garbage. We can do better.

---

## The weapon: hash set

Here's the idea. Grab a bowl. Any bowl.

Walk through the tokens one at a time. Before you drop each token in the bowl, peek inside and ask: **is this number already here?**

- If yes — duplicate. Alert the manager. Done.
- If no — drop the token in and move to the next one.

If you make it all the way through the stack without finding a match, there are no duplicates.

That's it. That's the whole pattern.

The bowl in this story is a **hash set**. In Python it's `set()`, in JavaScript it's `new Set()`, in Java it's `HashSet`. The rules across languages are the same: you can add items, you can check if an item exists, and both operations run in roughly **constant time**.

<Callout type="tip" title="Why 'constant time' matters">
A hash set doesn't scan through its contents when you ask "do you have this?". It jumps directly to the spot where that item would live and checks just that spot. Whether the set has 10 items or 10 million, the check takes about the same time. That is the magic.
</Callout>

---

## The code

Here's the two-line version in Python:

```python
def contains_duplicate(tokens: list[int]) -> bool:
    return len(set(tokens)) != len(tokens)
```

Read it like English. "If the set of tokens is smaller than the list of tokens, some of them collapsed into one — duplicates exist."

This is cute and it works, but it walks the entire list no matter what. Let's write the early-exit version, which is what you'd actually want on the job:

```python
def contains_duplicate(tokens: list[int]) -> bool:
    seen = set()
    for t in tokens:
        if t in seen:
            return True
        seen.add(t)
    return False
```

Four lines that do the work. Walk the list once, check before adding, bail the instant you find a repeat.

---

## Why is this so much faster?

Look at what changed.

| | Naive (compare all) | Hash Set |
|---|---|---|
| Time | O(n²) | **O(n)** |
| Extra space | O(1) | O(n) |
| Scales to 100k items? | Painfully | Instantly |
| Early exit? | Yes | **Yes** |

Yeah, you used more memory. The set grows as you walk. For 5,000 tokens, that's a few hundred kilobytes. On any computer built in the last 20 years, you don't care.

But the time savings are wild. For 100,000 tokens:

- Naive: ~5 billion operations
- Hash set: ~100,000 operations

That's a **50,000× speedup**. Same problem. Same computer. Just a smarter approach.

<Callout type="info" title="The tradeoff you just learned">
Most DSA improvements come down to this: spend a little memory to save a lot of time. The hash set is usually the first place you reach for that trade.
</Callout>

## Gotchas

**1. Forgetting the early exit.**
If you loop through the whole list even after finding a duplicate, you're wasting the speed advantage. Return as soon as you spot it.

**2. Adding before checking.**
If you add `t` to the set *before* you check, the set will always "contain" `t` when you look for it. You'll flag every single ticket as a duplicate of itself. Check first, add second.

**3. Trying to use this on non-hashable items.**
Hash sets work because the items have a stable hash. Numbers, strings, tuples — fine. Lists and dicts — not hashable in Python. If your items are complex objects, you need to hash them into something stable first.

**4. Assuming "constant time" means "instant".**
O(1) means "doesn't grow with input size". It doesn't mean "zero cost". A hash lookup is still slower than comparing two numbers. For 5 items, a nested loop might actually be faster in practice. For 5,000 or more, the hash set wins every time.

## The bigger lesson

You didn't just solve one problem. You learned the single most useful pattern in coding interviews:

**When a problem asks "have I seen this before?", reach for a hash set.**

That shows up in a surprising number of problems:

- Finding duplicates (this one)
- Checking if two lists share any element
- Detecting cycles in a linked list (spoiler: Dungeon 6)
- Counting unique things
- Sliding window problems that track "what's in my window right now"

Your brain should now have a new reflex. See "duplicate", "seen", "unique", "exists" — think hash set.
